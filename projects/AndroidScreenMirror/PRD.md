## 1. 产品概述

### 1.1 产品背景与目标

随着移动游戏云化、大屏办公、远程协作等场景的快速发展，用户对 Android 设备屏幕内容的实时投射需求日益强烈。现有方案（如 Google Cast、第三方投屏工具）普遍存在延迟高（>150ms）、依赖局域网路由、画质压缩严重等问题，影响用户体验。

本项目旨在实现一套**端到端超低延迟（<50ms）的 Android 屏幕镜像 SDK**，支持通过 **WiFi P2P 直连** 实现设备间无需路由器的直接通信，并提供**触摸反控**能力，使接收端可以反向控制发送端设备，适用于游戏操控、远程桌面、教育演示等场景。

核心目标：
- 端到端屏幕镜像延迟 < 50ms（网络延迟 < 10ms）
- 触摸反控延迟 < 10ms
- 支持 WiFi P2P 直连，无需路由器
- 支持硬件 H.264/H.265 编码，保障画质与性能
- SDK + App 双模块架构，支持第三方集成

### 1.2 核心功能列表

| 编号 | 功能模块 | 功能描述 |
|------|----------|----------|
| F01 | 屏幕采集 | 通过 MediaProjection / MediaCodec 采集屏幕帧 |
| F02 | 视频编码 | 硬件 H.264/H.265 编码，支持动态码率 |
| F03 | 视频解码 | 接收端低延迟解码渲染 |
| F04 | 网络传输 | 基于 UDP/RTP 的低延迟流媒体传输 |
| F05 | 触摸反控 | 编码传输触摸事件，接收端回注触摸 |
| F06 | WiFi P2P | 设备发现、连接建立与维护 |
| F07 | 音频采集（可选） | 捕获并传输设备音频 |
| F08 | 重连恢复 | 断线自动重连与状态恢复 |
| F09 | 带宽适配 | 动态调整码率适配网络状况 |
| F10 | 心跳与延迟检测 | 连接健康监测与延迟测量 |

### 1.3 目标用户与场景

| 场景 | 描述 | 典型用户 |
|------|------|----------|
| 游戏投屏 | 将手机游戏实时投射到大屏，支持触摸反控操控游戏 | 手游玩家、主播 |
| 大屏办公 | 手机屏幕投射到显示器/电视进行文档演示 | 商务人士、教师 |
| 远程桌面 | 远程查看并控制 Android 设备 | IT 运维、技术支持 |
| 教学演示 | 教师屏幕投射给学生屏幕，支持反向控制教学 | 教育机构 |
| 车机互联 | 手机内容投射到车机屏幕 | 车机厂商 |

### 1.4 产品定位（SDK + App）

本产品采用 **SDK + App 双模块架构**：

```
┌─────────────────────────────────────────────────────┐
│                    AndroidScreenMirror               │
├──────────────────────────┬──────────────────────────┤
│      screenshare-sdk      │       screenshare-app    │
│       （SDK 模块）         │        （演示 App）      │
│                          │                          │
│  • ScreenshareSDK        │  • 完整的投屏/接收演示    │
│  • Sender / Receiver      │  • UI 配置与状态展示     │
│  • VideoEncoder/Decoder   │  • 连接管理界面          │
│  • TouchEncoder/Decoder  │  • 触摸反控开关           │
│  • P2pConnectionManager  │                          │
│  • Network / Codec / ... │                          │
│                          │                          │
│  ▶ 供给第三方 App 集成     │  ▶ 提供完整用户体验      │
└──────────────────────────┴──────────────────────────┘
```

- **SDK 模块**（`screenshare-sdk`）：核心投屏能力封装，以 AAR 或 Maven dependency 形式供给第三方集成
- **App 模块**（`screenshare-app`）：完整的投屏演示应用，作为 SDK 的参考实现与测试载体

## 2. 功能需求

### 2.1 屏幕采集需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| CAP-001 | 支持通过 `MediaProjection` API 采集屏幕内容 | P0 |
| CAP-002 | 支持虚拟显示屏（Virtual Display）采集，解决 MediaProjection 隐私提示问题 | P1 |
| CAP-003 | 采集帧率可配置（目标 30fps / 60fps） | P0 |
| CAP-004 | 采集分辨率为设备原始分辨率，支持按比例缩放 | P0 |
| CAP-005 | 采集时序基于硬件 VSync，保证帧节奏稳定 | P1 |
| CAP-006 | 支持采集过程中动态切换采集源（多显示器场景） | P2 |

### 2.2 视频编码需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| ENC-001 | 支持硬件 H.264 编码（通过 MediaCodec Surface 输入） | P0 |
| ENC-002 | 支持硬件 H.265/HEVC 编码（设备支持时启用） | P1 |
| ENC-003 | 编码配置文件：Baseline / Main / High，默认 High | P0 |
| ENC-004 | 编码码率可动态调整（建议范围 1Mbps ~ 20Mbps） | P0 |
| ENC-005 | 支持 I 帧间隔配置（建议 2s ~ 5s） | P0 |
| ENC-006 | 支持零拷贝编码（Surface → MediaCodec，直接输入原始帧） | P0 |
| ENC-007 | 支持编码参数（码率、帧率、GOP）运行时动态更新 | P0 |
| ENC-008 | 编码失败时自动降级到软编码或降低分辨率 | P1 |

### 2.3 网络传输需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| NET-001 | 基于 UDP 传输视频流，支持 RTP 封装 | P0 |
| NET-002 | 视频帧分片传输，单帧可分割为多个 UDP 包 | P0 |
| NET-003 | 支持拥塞控制（GCC / NADA 等算法） | P1 |
| NET-004 | 支持 RTCP 反馈，用于报告接收质量 | P1 |
| NET-005 | 支持 jitter buffer 吸收网络抖动，缓冲时长可配置（建议 0ms ~ 100ms） | P0 |
| NET-006 | 支持多路并发连接（一台发送端对多台接收端） | P2 |
| NET-007 | 数据包支持序列号，用于丢包检测与重传请求 | P0 |

### 2.4 触摸反控需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| TOU-001 | 支持单指点击、滑动 | P0 |
| TOU-002 | 支持多指触控（最多 10 指） | P0 |
| TOU-003 | 触摸事件通过独立通道传输，优先级高于视频数据 | P0 |
| TOU-004 | 支持触摸坐标映射（接收端屏幕 → 发送端屏幕坐标转换） | P0 |
| TOU-005 | 触摸事件包含时间戳，用于精准重放 | P0 |
| TOU-006 | 支持 TouchKeyEvent（POWER、HOME、BACK 等系统按键） | P1 |
| TOU-007 | 支持长按、缩放（Gesture）事件识别与传输 | P1 |
| TOU-008 | 触摸反控注入基于 `AccessibilityService` 或 `InputManager` | P0 |

### 2.5 WiFi P2P 需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| P2P-001 | 支持 WiFi P2P 设备发现（discoverPeers） | P0 |
| P2P-002 | 支持请求连接与接受连接 | P0 |
| P2P-003 | 支持服务发现（NSD, Nearby Service Discovery）广播与监听 | P0 |
| P2P-004 | 连接状态机：Idle → Discovering → Connecting → Connected → Working →断开 | P0 |
| P2P-005 | 支持组所有者（Group Owner）角色协商 | P0 |
| P2P-006 | 连接建立后通过 socket 进行数据传输 | P0 |
| P2P-007 | 断开检测与自动重新连接 | P0 |
| P2P-008 | 兼容 WiFi P2P + 移动数据同时开启的场景 | P1 |

### 2.6 音频需求（可选）

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| AUD-001 | 支持通过 `AudioRecord` 采集设备输出音频（主音频流） | P1 |
| AUD-002 | 支持 Opus / AAC 音频编码 | P1 |
| AUD-003 | 音频与视频的时间戳同步（基于 AudioTimestamp） | P1 |
| AUD-004 | 接收端通过 `AudioTrack` 或 OpenSL ES 播放音频 | P1 |
| AUD-005 | 支持静音开关 | P1 |

### 2.7 重连与恢复需求

| 需求编号 | 需求描述 | 优先级 |
|----------|----------|--------|
| REC-001 | 检测连接断开后自动尝试重连（指数退避策略） | P0 |
| REC-002 | 重连期间保持采集编码，重连成功后快速恢复传输 | P0 |
| REC-003 | 支持发送端崩溃后接收端自动重连 | P0 |
| REC-004 | 支持接收端崩溃后发送端维持服务，等待重连 | P1 |
| REC-005 | 重连超时配置化（默认 30s，超时后通知 App） | P0 |
| REC-006 | 恢复传输时优先传输关键帧（I Frame）加速画面恢复 | P0 |

## 3. 性能需求

### 3.1 延迟指标

| 指标名称 | 目标值 | 说明 |
|----------|--------|------|
| 端到端延迟（E2E） | < 50ms | 从屏幕变化到接收端显示的总延迟 |
| 网络延迟 | < 10ms | 发送端到接收端的单程网络传输延迟（局域网） |
| 触摸反控延迟 | < 10ms | 从接收端触摸到发送端响应的延迟 |
| 编码延迟 | < 5ms | 视频帧编码耗时 |
| 解码延迟 | < 5ms | 视频帧解码耗时 |
| 端到端恢复时间 | < 2s | 断线重连后恢复画面的时间 |

### 3.2 画质指标

| 指标名称 | 目标值 | 说明 |
|----------|--------|------|
| 最大分辨率 | 1920×1080（向下兼容） | 支持根据终端能力动态调整 |
| 编码方式 | H.264 High Profile / H.265 Main Profile | 优先 H.265（同等画质下节省约40%码率） |
| 码率范围 | 1 Mbps ~ 20 Mbps | 自适应网络带宽 |
| 帧率 | 30fps / 60fps | 可配置，60fps 仅在网络条件良好时启用 |
| 色彩空间 | YUV420（NV21/YV12） | 与 MediaCodec 默认格式对齐 |

### 3.3 稳定性指标

| 指标名称 | 目标值 | 说明 |
|----------|--------|------|
| 运行时间 | ≥ 4 小时连续投屏无崩溃 | 验证内存泄漏与状态机健壮性 |
| CPU 占用 | ≤ 30%（发送端） / ≤ 15%（接收端） | 在 1080p@60fps 条件下测量 |
| 内存占用 | ≤ 200MB（发送端） / ≤ 150MB（接收端） | 不含 App 自身内存 |
| 帧率稳定性 | 帧率波动 ≤ 10% | 网络抖动情况下 |
| 功耗 | 发送端每小时耗电 ≤ 5% | 1080p@30fps 条件下 |

### 3.4 兼容性要求

| 要求 | 说明 |
|------|------|
| minSdk | API 26（Android 8.0 Oreo） |
| targetSdk | API 34（Android 14） |
| WiFi P2P | Android 8.0+ 内置支持 |
| 硬件编码 | 需要设备支持 MediaCodec 硬件编码（绝大多数设备支持） |
| 触摸反控 | Android 8.0+ 无特殊权限要求（AccessibilityService 安装即可） |
| 屏幕采集 | 需要用户授权 MediaProjection（一次性授权） |

## 5. 用户场景与流程

### 5.1 发送端场景

**发送端**指发起屏幕投射的一方（通常是 Android 手机/平板）。

```
┌─────────────────────────────────────────┐
│              发送端流程                  │
│                                         │
│  1. 初始化 SDK                          │
│         ↓                               │
│  2. 启动 P2P 服务发现                    │
│         ↓                               │
│  3. 被接收端发现/请求连接                │
│         ↓                               │
│  4. 用户授权 MediaProjection（首次）    │
│         ↓                               │
│  5. 建立 P2P 连接 → 开始屏幕采集编码     │
│         ↓                               │
│  6. 传输视频流（持续）                   │
│         ↓                               │
│  7. 接收触摸反控事件                     │
│         ↓                               │
│  8. 连接断开 / 停止服务                  │
└─────────────────────────────────────────┘
```

### 5.2 接收端场景

**接收端**指接收屏幕投射的一方（通常是 Android TV/机顶盒/另一个手机）。

```
┌─────────────────────────────────────────┐
│              接收端流程                  │
│                                         │
│  1. 初始化 SDK                          │
│         ↓                               │
│  2. 启动 P2P 服务发现并广播自身         │
│         ↓                               │
│  3. 发现发送端 → 请求连接               │
│         ↓                               │
│  4. 连接建立成功                        │
│         ↓                               │
│  5. 开始接收视频流 → 解码渲染           │
│         ↓                               │
│  6. 用户触摸屏幕 → 编码传输触摸事件     │
│         ↓                               │
│  7. 连接断开 / 停止服务                  │
└─────────────────────────────────────────┘
```

### 5.3 典型使用流程

```
  [发送端 Android 手机]              [接收端 Android TV]
         │                                  │
         │  ── P2P 服务发现 ──────────→    │
         │  ← ── 服务广播 ────────────     │
         │                                  │
         │  ── 连接请求 ─────────────→     │
         │  ← ── 接受连接 ────────────     │
         │                                  │
         │  [用户授权 MediaProjection]      │
         │                                  │
         │  ══ 屏幕采集 + 编码 ══════→     │  ══ 解码 + 渲染 ══
         │                                  │
         │  ← ── 触摸反控事件 ─────────     │  [用户触摸 TV 屏幕]
         │  ══ 触摸注入响应 ═════════→     │
         │                                  │
         │  ── 断开检测 ──────────────────  │
         │  ── 自动重连 ────────────────→   │
```

## 6. 竞品分析

| 竞品 | 延迟 | WiFi P2P | 触摸反控 | 编码方式 | 开放性 |
|------|------|----------|----------|----------|--------|
| Google Cast | ~150ms | ❌ 依赖局域网 | ❌ | H.264 | SDK 开放 |
| Scrcpy | ~35ms（有线）/ 100ms+（无线） | ❌ | ✅（有线） | H.264 | 开源（有线为主） |
| AirDroid | ~200ms | ❌ | ✅ | H.264 | 商业闭源 |
| LetsView | ~150ms | ❌ | ✅ | H.264 | 商业闭源 |
| **本项目 AndroidScreenMirror** | **< 50ms** | **✅** | **✅** | **H.264/H.265** | **SDK 开源** |

**竞争优势：**
- 超低延迟（< 50ms E2E）优于大多数竞品
- WiFi P2P 直连，无需路由器或互联网
- SDK + App 双模块，开放度高，支持第三方集成
- 支持硬件 H.265 编码，画质与码率兼优
- 触摸反控延迟 < 10ms，体验接近有线连接

## 7. 风险与约束

### 7.1 技术风险

| 风险编号 | 风险描述 | 缓解措施 |
|----------|----------|----------|
| R-01 | 部分设备硬件编码器不支持 H.265 | 代码中检测编码器能力，不支持时自动降级 H.264 |
| R-02 | WiFi P2P 在部分厂商定制 Android 上行为不一致 | 在主流设备（Pixel、三星、小米）上充分测试；抽象 P2P 层接口 |
| R-03 | 虚拟显示屏方案在 Android 10+ 权限收紧 | 跟进 Android 版本更新，优先使用标准 MediaProjection |
| R-04 | UDP 传输丢包导致花屏 | 实现 NACK 重传请求 + 关键帧快速重发机制 |
| R-05 | 长时运行内存泄漏 | 定期 review，自动化内存监控测试 |

### 7.2 商业/法律风险

| 风险编号 | 风险描述 | 缓解措施 |
|----------|----------|----------|
| R-06 | MediaProjection API 使用场景受限（Google Play 政策） | 仅用作屏幕镜像，符合 Google Cast 场景规范 |
| R-07 | 触摸反控涉及 AccessibilityService，可能被 Google Play 标记 | 提供免责说明，引导用户从官网下载而非 Play 商店 |

### 7.3 项目约束

| 约束类型 | 描述 |
|----------|------|
| 时间约束 | MVP 版本需在 8 周内完成 |
| 人力约束 | 核心开发 2~3 人 |
| 设备约束 | 需覆盖至少 5 款不同 Android 设备进行测试（含低端机型） |
| 技术约束 | minSdk=26，不支持 Android 8.0 以下版本 |
